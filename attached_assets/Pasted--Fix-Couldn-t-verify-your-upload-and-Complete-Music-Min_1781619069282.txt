# Fix “Couldn’t verify your upload” and Complete Music Minute Publishing

Do not add new features.

The previous `Storage unavailable` error has changed after the storage configuration fix.

The current physical-iPhone result is:

* Video selection works.
* The creator reaches Review and Post.
* Storage signing appears to work.
* The final error is:

`Couldn’t verify your upload. Please try again.`

This likely means the signed upload or confirmation phase is failing. Trace the exact request before changing code.

The required complete path is:

1. Request signed upload
2. Upload video bytes
3. Confirm stored object
4. Create Music Minute database row
5. Return published post
6. Show it in Home and Profile

## 1. Find the exact source of the message

Search for:

`Couldn’t verify your upload. Please try again.`

Report:

* Mobile file and function producing it
* Server endpoint called immediately before it
* HTTP status returned
* Structured error code returned
* Actual sanitized response body
* Whether the failure is in:

  * Storage verification
  * Authentication
  * Database insertion
  * Response parsing
  * Network timeout

Do not hide the real backend error behind the generic message.

## 2. Add one upload request ID

Generate one request ID at the start of posting and use it through:

* Sign request
* Direct storage upload
* Confirm request
* Database post creation

Log the request ID on mobile and server.

For the next real upload, report the last successful stage.

Do not log tokens, signed URLs, video bytes, or secrets.

## 3. Verify whether the video object was actually uploaded

For the failed attempt, check App Storage directly.

Confirm:

* Object exists
* Object key
* Object size is greater than zero
* Stored MIME type
* Stored filename extension
* Upload completion status
* Bucket/environment
* Whether the object belongs to the authenticated user’s upload session

Do not assume a successful mobile PUT response means the full file exists.

If the object does not exist, debug the direct upload.

If the object exists, debug the confirm endpoint.

## 4. Inspect the direct upload response

Record safely:

* HTTP method
* HTTP status
* Response headers
* Bytes expected
* Bytes sent
* MIME type sent
* Whether the upload URL expired
* Whether the phone added unexpected headers
* Whether the upload completed before confirm started

Confirm `.mov` uses `video/quicktime` and `.mp4` uses `video/mp4`.

Do not send an authentication header to the signed storage URL unless the signed-upload protocol explicitly requires it.

Do not change signed query parameters.

## 5. Fix object-key consistency

Verify that the exact same object key is used in all three places:

1. Returned by the signing endpoint
2. Used by the storage upload
3. Sent to the confirmation endpoint

Check for:

* Missing folder prefix
* Added leading slash
* URL encoding differences
* Filename changes
* Extension changes
* Development versus production path differences
* Confirm endpoint using a URL instead of the object key

The confirm endpoint should verify the stable object key generated by the server.

Do not trust an arbitrary object key supplied only by the client.

## 6. Verify private-storage behavior correctly

Determine how the confirm endpoint checks the object.

If App Storage is private, do not verify the upload by sending an unauthenticated `HEAD` request to a public-looking URL. That may return 403 even when the object exists.

Use the official App Storage mechanism or sidecar endpoint available to the API process to inspect:

* Object existence
* Size
* Content type
* Metadata

The upload-only signed URL must not be used as the permanent playback URL.

Store either:

* A stable object key served through an application media endpoint

or:

* A proper persistent readable URL appropriate for the bucket policy

## 7. Handle verification timing

The mobile app may call confirm immediately after the storage PUT finishes.

If storage verification can briefly lag, implement limited server-side retry:

* Retry object existence verification a small number of times
* Use short increasing delays
* Stop after a reasonable total duration
* Return a specific retryable error if still unavailable

Do not create the database post until a non-zero stored object is verified.

Do not use unlimited retries.

## 8. Inspect the confirm endpoint

Confirm that the endpoint:

* Requires valid Clerk authentication
* Receives the expected request shape
* Finds the upload session
* Verifies the correct authenticated user
* Verifies the object exists
* Verifies size is greater than zero
* Verifies supported MIME type
* Does not reject valid iPhone `.mov` files
* Does not depend on a temporary signed URL
* Returns structured JSON

Possible controlled error codes:

* `UPLOAD_SESSION_NOT_FOUND`
* `UPLOAD_SESSION_EXPIRED`
* `UPLOAD_OBJECT_NOT_FOUND`
* `UPLOAD_OBJECT_EMPTY`
* `UPLOAD_CONTENT_TYPE_MISMATCH`
* `UPLOAD_OWNER_MISMATCH`
* `UPLOAD_CONFIRM_AUTH_FAILED`
* `POST_CREATE_FAILED`

Never collapse all of these into one generic verification failure in development logs.

## 9. Separate verification from database creation

Determine whether “Couldn’t verify your upload” is masking a database failure.

The correct stages are:

### Stage A — Storage verification

Confirm object exists and has valid metadata.

### Stage B — Post creation

Insert the Music Minute row with:

* Creator ID
* Storage object key
* Playback URL or media route
* MIME type
* File size
* Duration
* Title
* Type
* Genre
* Language
* Musixmatch track ID
* Song title and artist
* Lyric section label
* Line indexes
* Timing mode
* Timestamps when real
* Timing offset

Return different errors for Stage A and Stage B.

If verification succeeds but database creation fails, do not ask the phone to upload the video again. Allow retrying only the post-save stage.

## 10. Check the active environment

Report which environment the physical phone is using:

* Development or published API
* Development or production PostgreSQL
* Development or production App Storage context

Confirm sign, upload, confirm, and database creation all use the same environment.

Do not sign against one storage environment and confirm against another.

## 11. Improve the mobile flow

Display distinct stages:

1. Preparing video
2. Uploading video
3. Verifying upload
4. Saving Music Minute
5. Published

If verification fails but the object exists, preserve the upload session and retry confirmation without uploading the whole video again.

Retry must not:

* Create a duplicate object
* Create duplicate posts
* Start two concurrent uploads
* Reuse an expired upload URL incorrectly

Run physical-iPhone tests in this order:

### Test 2

For each test report:

## 14. Acceptance criteria

Do not mark this fixed until:

* The exact cause of verification failure is known.
* The uploaded object exists and is non-zero.
* Confirm checks the correct object key.
* Private storage is verified through the correct server mechanism.
* Confirmation and database creation have separate errors.
* A 5-second, 20-second, and 53-second iPhone video publish successfully.
* The post appears in Home and Profile.
* Playback works after restart.
* Retry does not create duplicates.
* Mobile and API-server typechecks pass.
