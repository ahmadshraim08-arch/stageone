# Priority 2 — Fix `spawn ffmpeg ENOENT`

The analysis screen currently fails with:

`Audio extraction failed: spawn ffmpeg ENOENT`

This means the runtime attempted to launch FFmpeg but could not find the executable.

## Find the exact code path

Report:

* File and function calling `spawn`
* Command being executed
* Executable path used
* Arguments
* Development runtime PATH
* Published runtime PATH
* Whether FFmpeg exists in each runtime
* Whether the failure occurs in development, deployment, or both

Do not expose the raw technical error to normal users.

## Preferred solution: avoid unnecessary extraction

Inspect the selected transcription or audio-analysis provider.

If it accepts the original video format directly:

* Send the stabilized video file directly
* Skip FFmpeg extraction
* Preserve MIME type and filename
* Do not convert the video to Base64

This is preferred when supported because it removes an unnecessary failure point.

## If audio extraction is required

Make FFmpeg available reliably in both development and deployed runtimes.

Use one supported approach:

* Install/configure FFmpeg as a persistent Replit system dependency
* Bundle a compatible static FFmpeg binary through a package
* Resolve the binary path explicitly rather than assuming `ffmpeg` is in PATH

Do not fix it only in the temporary shell.

Before processing, run a startup health check:

* Resolve FFmpeg executable
* Run `ffmpeg -version`
* Report `audioExtraction: "ok"` or a specific failure code in diagnostics

Use the resolved absolute executable path in `spawn`.

## Extraction requirements

Create a temporary audio analysis file with:

* Mono audio where suitable
* A transcription-friendly sample rate
* A supported audio format
* Original video preserved unchanged

After extraction verify:

* Process exit code is 0
* Output file exists
* File size is greater than zero
* Duration is valid
* MIME type is correct

Then:

* Send it to the transcription provider
* Delete it when analysis finishes or fails
* Do not store it permanently
* Do not put it in PostgreSQL
* Do not read it as Base64

## Error handling

The user should see:

**We couldn’t analyze the audio. You can retry or search for the song manually.**

Development logs should retain codes such as:

* `FFMPEG_NOT_AVAILABLE`
* `AUDIO_EXTRACTION_FAILED`
* `AUDIO_OUTPUT_EMPTY`
* `TRANSCRIPTION_FAILED`

Do not show `spawn ffmpeg ENOENT` in the mobile UI.

# Priority 3 — Automatically suggest language and genre

After analysis and song identification, Step 4 should be pre-filled intelligently.

The user must still be able to override both values.

## Automatic language detection

Use this priority:

1. Transcription service detected language
2. Musixmatch lyric language or track-language metadata
3. Existing creator selection
4. Manual selection

Save:

* Suggested language
* Detection source
* Confidence when available
* Whether the creator manually changed it

Display:

**Detected: English**

Allow the creator to tap another language.

If transcription and Musixmatch disagree, do not silently choose. Show the most reliable suggestion and allow correction.

## Automatic genre detection for covers

After the real Musixmatch track is identified:

* Inspect official track genre metadata returned by the API
* Map the official genre to StageOne’s available genre options
* Preselect the closest valid StageOne genre
* Preserve the original API genre in diagnostics
* Allow manual override

Example mapping may normalize values such as:

* Contemporary R&B → R&B
* Hip-Hop/Rap → Rap
* Singer/Songwriter → Acoustic or Pop, depending on available categories

Do not invent genre based solely on lyric text.

## Original and Freestyle handling

For Original or Freestyle posts, Musixmatch track metadata may not exist.

Use:

* A configured audio-analysis partner such as Cyanite, if available and authorized

Otherwise:

* Keep genre manual
* Do not falsely label it as automatically detected

The feature must clearly distinguish:

* `musixmatch_metadata`
* `audio_analysis`
* `creator_selected`
* `unknown`

## Step 4 experience

When Step 4 opens, show suggestions such as:

* **Genre detected: Pop**
* **Language detected: English**

Preselect the suggested chips.

The creator can change them before publishing.

Do not block publishing when automatic detection fails.

## Persist detection metadata

Store minimal fields such as:

* `genre`
* `genreDetectionSource`
* `genreConfidence`
* `language`
* `languageDetectionSource`
* `languageConfidence`
* `creatorOverrodeGenre`
* `creatorOverrodeLanguage`

Do not permanently store full transcripts unless required and permitted.
