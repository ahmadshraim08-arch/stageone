# Fix “Storage unavailable” and Complete the Real Music Minute Upload Cycle

Do not add new features.

The video picker now works and the user reaches the final Review and Post screen. The current blocking error is:

`Storage unavailable. Please try again.`

This prevents every real Music Minute from being uploaded and blocks full app testing.

Important architecture requirement:

* Store the actual video file in Replit App Storage.
* Store only the video reference and post metadata in PostgreSQL.
* Do not save the complete video binary or Base64 content inside the database.

The production database is available and has sufficient capacity for post records and metadata, but the video file must be uploaded to App Storage and then linked to its database post.

## Immediate objective

Complete this real end-to-end path:

1. Physical iPhone selects video
2. App prepares local file
3. Mobile requests an upload session or signed upload URL
4. API verifies the authenticated user
5. Replit App Storage accepts the video
6. Storage returns an object key or persistent media URL
7. API creates the Music Minute database record
8. Post appears in Home
9. Post appears on creator Profile
10. Post plays after restarting the app

Do not stop after fixing only the error message.

---

# 1. Find the exact origin of “Storage unavailable”

Search the complete codebase for the exact text:

`Storage unavailable. Please try again.`

Report:

* File and function that produces it
* API request that happens immediately before it
* Expected response shape
* Actual response shape
* HTTP status
* Server log connected to that request
* Whether the failure occurs before an App Storage upload URL is created
* Whether the mobile app receives an empty or malformed upload configuration
* Whether the message is hiding a more specific backend error

Do not expose secrets, tokens, or private signed URLs.

Replace vague logging with a sanitized technical error code such as:

* `STORAGE_NOT_CONFIGURED`
* `STORAGE_SIGNING_FAILED`
* `STORAGE_BUCKET_UNAVAILABLE`
* `UPLOAD_SESSION_FAILED`
* `STORAGE_PERMISSION_DENIED`

The user can still see a friendly message, but development logs must reveal the real cause.

# 2. Verify that App Storage exists and is accessible

Inspect the Replit project’s App Storage configuration.

Confirm:

* An App Storage bucket exists for this application
* The current development runtime can access it
* The published deployment can access it
* The API server uses the correct bucket
* Required App Storage environment configuration exists in the running process
* The App Storage SDK or signing mechanism initializes successfully
* Development and production are not pointing to incompatible buckets or missing configuration
* The API server was restarted after storage configuration changed

Do not assume configuration is valid because an environment-variable name exists.

Run a real server-side storage health test:

1. Upload a small text object
2. Confirm it exists
3. Read it back
4. Delete it
5. Report every result

Then run a small binary-object test.

If either test fails, fix App Storage configuration before continuing with mobile video upload.

# 3. Separate Database and App Storage responsibilities

Use this architecture:

## App Storage stores

* Original uploaded video
* Optional future thumbnail
* Optional profile images

## PostgreSQL stores

* Music Minute post ID
* Creator user ID
* Video object key
* Persistent video URL or media-serving route
* MIME type
* File size
* Duration
* Width and height
* Title
* Performance type
* Genre
* Language
* Musixmatch track ID
* Song title
* Artist
* Lyric section label
* Start/end line indexes
* Start/end timestamps when available
* Timing mode
* Timing offset
* Created timestamp
* Processing or publication status

Do not place the full video byte content into PostgreSQL.

Do not store a local iPhone URI in PostgreSQL.

Do not store a temporary signed upload URL as the permanent video URL.
