# Fix the Complete Musixmatch Feature Set Before Testing

Do not add unrelated features.

The Musixmatch API key is now authenticating correctly. Real track search, track metadata, and lyric translation are working.

The remaining broken behavior is:

* Synced subtitles return an empty body for some real tracks.
* The lyrics route returns `lines: []`.
* Section selection shows an invalid `Selected Section 0–0`.
* “Sing This Part” is blocked for real songs.
* RichSync is unavailable for the tested track or API access level.
* Mood analysis currently falls back to locally derived genre data.

The objective is to make the complete StageOne Musixmatch flow usable before physical-device testing:

**Search real song → load usable lyrics → select a real section → publish → display lyrics → translate → tap Sing This Part.**

Do not pretend unsupported Musixmatch features are available. Use honest fallbacks and clearly label the source.

---

# 1. Diagnose every Musixmatch response first

Test at least five real tracks with different characteristics:

* A popular English song
* A song known to have synced subtitles
* A song with translations
* A recent song
* A song from another language

For each track test:

1. `track.get`
2. `track.lyrics.get`
3. `track.subtitle.get`
4. `track.richsync.get`
5. Translation endpoint
6. Lyrics analysis or mood endpoint

Record:

* HTTP status
* Musixmatch status code
* Whether the body exists
* Whether the body is empty
* Number of lines
* Number of timed lines
* Number of timed words
* Available translations
* Whether the endpoint appears restricted by the API plan

Do not expose the API key.

Do not assume that metadata such as `has_subtitles: 1` guarantees a non-empty subtitle body.

---

# 2. Repair the lyrics fallback chain

Update the server-side lyrics resolver to use this exact order:

## First: RichSync

Attempt RichSync.

Use it only when:

* The request succeeds
* The response body exists
* Parsed lines exist
* Parsed words contain valid timing data

If RichSync returns an error, restricted access, an empty body, malformed JSON, or zero usable words, continue to subtitles.

## Second: synced subtitles

Attempt synced subtitles.

Use them only when:

* The subtitle body exists
* Parsing produces at least one non-empty lyric line
* Valid timestamps exist

Do not return:

* `hasSync: true` with `lines: []`
* `source: "musixmatch"` merely because metadata says subtitles exist

If subtitle parsing produces no lines, continue to plain lyrics.

## Third: plain lyrics

Call the plain lyrics endpoint.

Parse the lyric body into non-empty lines.

Return:

* `hasSync: false`
* `hasRichsync: false`
* Real lyric lines
* Correct Musixmatch attribution
* `source: "musixmatch"`

Remove Musixmatch footer markers or legal boilerplate only when permitted and while preserving the required copyright text separately.

## Fourth: demo fallback

Use demo lyrics only when all real Musixmatch lyric methods fail.

Return:

* `source: "demo"`
* `hasSync: false` unless real demo timing exists
* A visible Demo Content label in the mobile app

Never label demo lyrics as Musixmatch data.

---

# 3. Normalize the lyrics API response

Return one stable response shape regardless of source:

```text
{
  trackId,
  source,
  mode,
  hasSync,
  hasRichsync,
  language,
  copyright,
  lines: [
    {
      index,
      text,
      startMs,
      endMs,
      words
    }
  ]
}
```

Use `mode` values such as:

* `richsync`
* `subtitle`
* `plain`
* `demo`

For plain lyrics:

* `startMs` and `endMs` should be `null`
* `words` may be empty
* `hasSync` must be false

Do not create fake timestamps for plain lyrics.

The mobile app must not crash when timing values are null.

---

# 4. Fix section selection for plain and synced lyrics

The section picker must work even when RichSync and subtitles are unavailable.

## When timed lyrics exist

Create sections from real timing data.

Allow the creator to choose:

* A group of lyric lines
* A start time
* An end time

Do not return zero-length sections.

## When only plain lyrics exist

Allow manual line-based section selection.

The user should be able to choose a contiguous group of lines.

Store:

* Start line index
* End line index
* Section label
* `startMs: null`
* `endMs: null`
* `timingMode: "manual"`

Use labels such as:

* Selected Section
* Opening Lines
* Middle Section
* Closing Section

Do not show:

* `0–0`
* Empty section
* Invalid timestamps
* A section containing no lyric text

Require at least one selected line before continuing.

## When no lyrics exist

Disable lyric-section selection and show:

**Lyrics are unavailable for this track. Choose another song or continue without LyricStage lyrics.**

Do not silently create a broken section.

# 5. Fix the segment endpoint

The segment endpoint must never return an invalid placeholder.

Required behavior:

* RichSync available → timed segments
* Subtitles available → timed line-based segments
* Plain lyrics available → line-index segments without timestamps
* No lyrics available → empty segments array plus a clear reason

Return metadata such as:

```text
{
  source,
  timingMode,
  segments,
  reason
}
```

Do not return a segment with start and end both equal to zero unless it is genuinely valid.

# 13. Correct source labeling

Every endpoint must return one of:

* `musixmatch`
* `derived`
* `demo`
* `unavailable`

Rules:

* Real API body parsed successfully → `musixmatch`
* Locally calculated result → `derived`
* Local sample content → `demo`
* No usable result → `unavailable`

Never infer the source from track metadata alone.

Base the source on the actual content returned.
